PHASE 01

Populate the images

Est. 1–2 days · designer + dev

The template renders every image slot as a <Placeholder> primitive (see components/Primitives.tsx) with a tiny mono caption. Phase 1 is mechanical: inventory the slots, source/shoot/license imagery, drop into /public/images/, swap the component for next/image.

1.1Audit

Inventory every placeholder slot

Have Claude Code grep the codebase and produce a typed manifest of every image position, its aspect ratio, page, and a one-line content brief. This becomes the shot-list for the photographer / stock-image search.

1.2Source

Decide treatment per slot & gather assets

Forensic imagery is genre-sensitive — wrong photos look like stock-y "cybersecurity" sludge. Pair Claude Code with explicit creative direction:

PHOTO

Studio shoot

Founders, lab interior, evidence workstation, classroom. Brief a photographer with the manifest. Output 2400px JPEG / WebP.

EDITORIAL

Stock with treatment

Sector "partnership" shots, insights tiles. Pull from licensed library (Unsplash+ / Getty), apply consistent green-grade LUT.

SVG

Icons & textures

Value-pillar icons, ph-grid backdrops, abstract case-file headers. Keep house-style: 1px stroke, accent green on ink.

[ Creative guard-rail ]

Cyber-Eye is a forensics firm, not a SaaS. Avoid: hooded hackers, padlock icons, binary rain, blue HUD overlays. Prefer: hands on hardware, chain-of-custody envelopes, real Cellebrite/FTK rigs, courtroom benches, lab benches at night.

1.3Process

Optimise & place into /public/images/

One folder per route; one filename per id from the manifest. Run a single script to produce WebP + AVIF + a tiny blur placeholder. Claude Code can author and run the pipeline:

Claude Code prompt→ scripts/optimise-images.ts
claude "Write a Node script at scripts/optimise-images.ts that: 1. Reads every file in /raw/ (input folder) 2. Uses sharp to emit 3 variants per source: 1920w .webp, 960w .webp, 480w .webp 3. Generates a base64 blurDataURL (10x10, blurred) 4. Writes the blur strings to lib/image-blur.ts as a typed map keyed by manifest id 5. Drops outputs into /public/images/[route]/[id]-[width].webp Add an npm script: 'images' that runs it. Idempotent."
1.4Swap

Replace <Placeholder> with next/image

One mechanical pass per route. Keep the existing aspect-ratio className — only the inner element changes. Have Claude Code do it route by route and produce a single PR per phase chunk so review stays sane.

Claude Code prompt→ refactor pass
claude "For every <Placeholder ... /> in app/, replace with <Image /> from next/image. Rules: - Look up the slot's id, src and blurDataURL in lib/image-manifest.ts + lib/image-blur.ts. - Use the manifest's label as the alt text (rewrite to be descriptive, not the mono caption). - Preserve every className from the original Placeholder. - Keep object-cover; add sizes='(max-width: 768px) 100vw, 50vw' for half-width slots and 100vw for full-bleed. - For circular avatar slots (w-10 h-10 rounded-pill), use width/height 40. Open one PR per route group: home, about, services, sectors, case-files, insights, training."
1.5Verify

Visual regression + Lighthouse

  • Snapshot every route at three viewports (375 / 1024 / 1440) using Playwright; commit baselines.
  • Run Lighthouse per route — LCP < 2.5s on a throttled 4G profile is the gate.
  • Check that ph-grid backdrops still render where they're intended (CTA section on home, training hero, services CTA) — those are decorative, not placeholders.
PHASE 02

Backend & production work

Est. 2–3 weeks · 1 senior dev

The frontend is dummy-content-driven. To ship, we replace lib/data.ts with a real CMS, wire the contact form to a real intake pipeline, add SEO / sitemaps / structured data, lock down security, and deploy. Order matters — do not skip the staging environment.

2.1Content

Move lib/data.ts to a real CMS

Five collections (Services, Sectors, Case Files, Courses, Articles) are currently static TypeScript arrays. Pick a headless CMS — Sanity or Payload are the two clean fits for this codebase — then have Claude Code generate the schema and a typed fetch layer.

PICK A

Sanity

Hosted, fast editor UX, GROQ queries. Best if a non-dev edits content. ~$15/editor/mo.

PICK B

Payload CMS

Self-hosted, TypeScript-native, lives in the same Next.js repo. Best if you control infra.

PICK C

MDX in-repo

Skip CMS; treat case-files / insights as MDX in /content. Cheapest, dev-only edits.

Claude Code prompt→ Sanity scaffold
claude "Scaffold a Sanity Studio inside the repo at /studio. For each type in lib/data.ts (Service, Sector, CaseFile, Course, Article), create a matching Sanity schema preserving every field. Add a richText body field where currently there is none (Service, CaseFile, Article). Create lib/sanity.ts with a typed client and one helper per collection (e.g. getServices, getServiceBySlug). Replace every import of lib/data.ts in app/** with the Sanity helper, using await + ISR (revalidate: 300). Add a /studio route mounting the embedded Studio."
2.2Intake

Wire the /contact form to a real intake

Today the form has no action. For a forensics firm, intake is the most sensitive endpoint on the site — treat it accordingly.

→ STEP AConvert to a React Server Action with Zod validation on the input shape.app/contact/page.tsx
→ STEP BOn submit: write a record to a private table (Postgres / Supabase / Notion DB).DB
→ STEP CFire a transactional email via Resend to intake@cyber-eye.in with the matter summary.Resend
→ STEP DReply-acknowledgement to the requester with NDA attached as PDF.Resend
→ STEP EBot protection: Cloudflare Turnstile or hCaptcha + rate-limit by IP via Upstash.Edge
→ STEP FAudit log every intake (request id, IP, UA, hash). Retention 2 years per § 65B prep.Audit
[ Security gate ]

Do not store the matter narrative in plain-text in a third-party tool. Encrypt the matter field at rest using a KMS-held key; only decrypt server-side for the intake email.

Claude Code prompt→ contact intake
claude "Wire /contact end-to-end: 1. Convert the form to a Next.js Server Action submitContact(input). 2. Add Zod schema in lib/schemas/contact.ts; reject if matter < 20 chars or email invalid. 3. Add Cloudflare Turnstile widget; verify token server-side. 4. Rate-limit 3 submissions / IP / hour with @upstash/ratelimit. 5. Persist to Supabase table 'intakes' (encrypted matter via libsodium sealed box). 6. Send two emails via Resend: internal (intake@cyber-eye.in) + acknowledgement. 7. Return Server-Action state to the form for success/error rendering. Add .env.example with every key required."
2.3SEO

Sitemaps, structured data, OG, llms.txt

  • app/sitemap.ts — dynamic, sources URLs from Sanity (services, sectors, case files, articles).
  • app/robots.ts — allow all except /studio and /api/*.
  • JSON-LD generators for: Organization (root layout), Service (per service page), Article (per insights post), FAQPage (services FAQ), BreadcrumbList (every detail page).
  • OG image generator at app/og/[slug]/route.tsx using @vercel/og; renders the headline on the Cyber-Eye dark template.
  • /llms.txt at root listing services, sectors, contact — for AI crawlers per the wireframe spec.
Claude Code prompt→ SEO bundle
claude "Add the full SEO surface: - app/sitemap.ts (dynamic, ISR 1h) - app/robots.ts - lib/jsonld.ts with typed factories: orgLd, serviceLd, articleLd, faqLd, breadcrumbLd - Inject the right LD+JSON <script> tag in each page's metadata - app/og/[slug]/route.tsx with @vercel/og — dark BG, accent green corner, headline in Inter 500 - public/llms.txt summarising the site for AI discoverability - Per-page generateMetadata() in every dynamic route, pulling title/description from Sanity."
2.4Telemetry

Analytics, error monitoring, uptime

ANALYTICS

Plausible or Vercel Analytics

Cookie-less. Track: route views, CTA clicks (contact / quote / training enrol), outbound links.

ERRORS

Sentry

Client + server. Source-maps uploaded on build. Alert on 5xx spike, intake form failure rate > 2%.

UPTIME

BetterStack / UptimeRobot

Synthetic check every minute on / and /contact. Status page at status.cyber-eye.in.

2.5Hardening

Security, headers, compliance

  • Strict CSP via next.config.js headers — script-src self + nonce; no 'unsafe-inline'.
  • HSTS, X-Frame-Options DENY, Referrer-Policy strict-origin-when-cross-origin, Permissions-Policy disable camera/mic/geo.
  • Dependabot + npm audit in CI; fail builds on high+ CVEs.
  • Privacy & Terms (/legal/[slug]) — replace template copy with counsel-reviewed text.
  • India: DPDP Act 2023 compliance note (data fiduciary, contact, grievance officer) at the footer + privacy page.
  • Cookie banner only if analytics provider sets cookies — Plausible avoids the banner entirely.

2.7Launch

Go-live checklist

  • All Phase-1 images live; Lighthouse > 90 on every route.
  • CMS content reviewed by founder + legal counsel.
  • Contact form tested end-to-end with a real submission; ack email received; record encrypted in DB.
  • Search Console + Bing Webmaster Tools verified; sitemap submitted.
  • Backups verified — restore one from Supabase to a staging DB.
  • Status page public. Runbook for the most likely incidents (intake outage, CMS publish failure) in RUNBOOK.md.
  • DNS cut-over scheduled for a Tuesday morning IST; rollback plan documented.

Working with Claude Code

conventions for this repo
RULE 1

One PR per step

Every numbered step above maps to one PR. Keeps reviews surgical and reverts cheap.

RULE 2

Manifests, not magic

Anything Claude generates that touches many files (image manifest, JSON-LD, sitemap) must read from a single typed source — never inline.

RULE 3

Branch protection on

main requires PR + green CI + 1 approval. No direct pushes — including from Claude.

[ Repo hygiene ]

Add a CLAUDE.md at the repo root with these rules + a pointer to lib/image-manifest.ts and the Sanity schema. Claude Code reads it on every session and stays inside the system.
